# MCP & Skills 学习笔记

> 目标：理解 MCP、Skills、Commands、Plugins 的原理，从使用者进阶到创造者。

---

## 核心概念速览

### 概念关系图

```
你的 Agent 项目
      │
      ├── 需要"能力"（能做什么）  →  Tools / MCP Server
      │
      └── 需要"工作流"（怎么做）  →  Prompts / Commands / Skills
```

### 名词对照表

| 概念 | 本质 | 类比 |
|------|------|------|
| **Tool** | 一个可被 LLM 调用的函数 | 你在 mini-claw/tools.py 里写的那些 |
| **MCP Server** | 封装了若干 Tool 的独立服务 | 微服务 |
| **MCP Client** | 连接并使用 MCP Server 的一方 | Claude Code 本身就是 MCP Client |
| **Command** | 本地自定义的 prompt 快捷方式 | ~/.claude/commands/*.md |
| **Skill** | 打包发布后的 Command | npm 包里的一个函数 |
| **Plugin** | 包含若干 Skill 的发布单元 | npm 包本身 |

---

## Step 1 — 纯 Prompt Command

### ⭐ 核心结论

**Command/Skill 的本质：有名字的可复用 prompt 模板。**

没有任何逻辑、没有代码执行、没有参数解析 — 纯文本的快捷方式。

### 工作原理

```
用户输入：  /my-agent-review 检查 mini-claw

Claude Code 处理：
  1. 识别 / 开头
  2. 找到 ~/.claude/commands/my-agent-review.md
  3. 把 $ARGUMENTS 替换成 "检查 mini-claw"
  4. 把整段拼好的文字发给 LLM
```

等价于手动输入了整个 md 文件的内容。省去的是重复写长 prompt 的工作。

### 文件结构

```markdown
---
description: 这里写 command 的描述，会显示在下拉菜单里
---

这里是 prompt 正文...

$ARGUMENTS
```

- `description`：frontmatter，显示在 `/` 下拉菜单的提示里
- `$ARGUMENTS`：占位符，被替换成用户在 `/command` 后面输入的所有文字
- `$ARGUMENTS` 不是必须的，不需要用户传参数时可以省略

### 存放位置

| 位置 | 作用域 |
|------|--------|
| `~/.claude/commands/` | 全局，所有项目都能用 |
| `<项目>/.claude/commands/` | 仅当前项目 |

文件名 = 命令名。`my-agent-review.md` → `/my-agent-review`

### 命名建议

加前缀避免和 Claude Code 内置命令冲突，例如 `/my-xxx`。

### 关于参数

- 只有一个 `$ARGUMENTS`，是 `/command` 后面所有文字的整体字符串
- 没有 `$ARG1`、`$ARG2` 的多参数机制
- 如需多参数，在 prompt 里说明格式，靠 LLM 自行解析

### Skills 与 MCP 的关系

```
Skills（工作流层）  +  MCP（能力层）
      │                    │
  告诉 Claude          给 Claude
  "怎么做"             "做什么的工具"
```

Skill 是指挥官，MCP 是执行工具。两者独立，也可叠加。

### 在自己的 Agent 项目里实现同等机制

Claude Code 帮你实现了触发和拼接。自己的项目里需要自己写：

```python
def handle_message(message):
    if message.startswith("/"):
        parts = message.split(" ", 1)
        skill_name = parts[0][1:]
        user_args = parts[1] if len(parts) > 1 else ""

        skill_content = open(f"skills/{skill_name}.md").read()
        message = skill_content.replace("$ARGUMENTS", user_args)

    return agent.run(message)
```

核心就是文本替换，没有更多。

### 价值

- 不用每次重复写长 prompt
- 团队可以共享统一的工作流
- 打包成 Plugin 可以分发给别人

## Step 2 — MCP Server

### ⭐ 核心结论

**MCP Server 的本质：把 Tool 标准化成独立服务，任何 MCP Client 都能复用。**

没有 MCP 时，每个 agent 项目都要重写 tool 的 schema + 实现 + dispatch。
MCP 之后，写一次，到处用。类比微服务。

### MCP Server 的必要条件

```
1. 官方 SDK（pip install mcp）
2. list_tools handler   — 告诉 client 有哪些 tool
3. call_tool handler    — 实际执行 tool 的地方
4. 通信入口             — stdio 或 HTTP
```

缺一不可。最小结构：

```python
server = Server("my-server")

@server.list_tools()
async def list_tools(): ...       # client 连接时调用，构建 tool 列表

@server.call_tool()
async def call_tool(name, args): ...  # client 调用 tool 时进入

async with stdio_server() as (r, w):
    await server.run(r, w, ...)   # 启动通信
```

### 通信方式

| 方式 | 适用场景 |
|------|---------|
| stdio | 本地 server，最常见，Claude Code 默认用这个 |
| HTTP/SSE | 远程 server，需要网络访问 |

stdio 就是标准输入输出，不需要端口，不需要 HTTP，进程间直接通信。

### 配置文件体系

```
.mcp.json                    ← 定义有哪些 server、如何启动（commit 进 git）
.claude/settings.json        ← Claude Code 行为配置（commit 进 git）
.claude/settings.local.json  ← 个人信任决定，自动生成（加入 .gitignore）
```

**优先级（从高到低）：**
```
项目 settings.local.json
项目 settings.json
用户 settings.local.json  (~/.claude/)
用户 settings.json        (~/.claude/)
```

`.mcp.json` 定义 server 存不存在，`settings.json` 控制连不连。

### 作用域

| 位置 | 作用域 | 是否 commit |
|------|--------|------------|
| 项目根目录 `.mcp.json` | 仅当前项目 | ✅ 应该 |
| `~/.claude/` 下配置 | 全局所有项目 | 个人决定 |

### ⚠️ 重要观察：MCP tool 的可见性

MCP tool 的可用性完全由 server 连接状态决定：

- **Server 连上** → tool 出现在 LLM 的可用 tool 列表里，LLM 能自主调用
- **Server 断开** → tool 从列表消失，LLM 无法调用

**陷阱：** 如果 server 曾经连上、用过、再断开，对话 history 里还有调用记录。LLM 会"知道"这个 tool 存在但无法使用，行为和从未连过不同。

→ 实际开发中，agent 依赖的 MCP server 挂掉时应明确报错，不能让 LLM 在"半知道"状态下继续运行。

### 什么时候用 MCP，什么时候直接写函数

```
功能只用一次，只服务一个 agent  →  直接写函数，别过度设计
功能需要跨多个 agent/应用复用   →  MCP Server
功能需要独立部署、独立更新      →  MCP Server
功能是第三方提供的              →  MCP Server
```

### Command vs Tool vs System Prompt

| | 触发方 | 适用场景 |
|--|--------|---------|
| Command/Skill（`/xxx`） | 人工触发 | 固定工作流，人决定何时用 |
| Tool / MCP tool | LLM 自主判断 | agent 自动决定何时需要某个能力 |
| System Prompt | 自动，每次对话 | 固定的背景指令 |

**LLM 不会自己去"发现"并调用 Command**，Command 只能人工触发。
想让 LLM 自动使用某个能力，就要用 Tool，不是 Command。

## Step 3 — Command + MCP 联动

### ⭐ 核心结论

**Command + MCP 就是把 Step 1 和 Step 2 拼在一起，没有新的技术点。**

```
用户输入 /my-recall 看全部
    ↓
Command 展开 prompt，告诉 LLM "你有记忆库工具，根据请求选择调用"
    ↓
LLM 判断需要调用 list_memories
    ↓
MCP Server 执行，返回结果
    ↓
LLM 用自然语言总结给用户
```

### Command 如何指示 LLM 调用 MCP tool

Command 的 prompt 里直接用自然语言描述何时调用哪个 tool：

```markdown
- 如果用户想查看全部记忆 → 调用 `list_memories`
- 如果用户想搜索         → 调用 `search_memory`
- 如果用户想保存         → 调用 `save_memory`
```

LLM 根据这段指示 + 可用 tool 列表，自己决定调哪个。**Command 不直接调用 tool，它只是给 LLM 下指令。**

### 关于 Command 的作用域

| 位置 | 作用域 |
|------|--------|
| `~/.claude/commands/` | 全局，所有项目 |
| `<项目>/.claude/commands/` | 仅当前项目 |

通用工作流（如代码审查）放全局；依赖特定 MCP Server 的 command 放项目级，因为 MCP Server 也是项目级的。

## Step 4 — 层级 Skill（Hierarchical Skill）

### ⭐ 核心结论

**层级 skill 不是 Claude Code 的原生功能，而是一种 prompt 设计模式。**

顶层 skill.md 用自然语言指示 Claude 按需读取子模块文件，Claude 利用内置的 Read tool 执行读取。框架不会自动扫描子目录，是 Claude 根据 prompt 指令主动去读的。

### 结构

```
my-agent-audit/
├── skill.md          ← 顶层入口：描述有哪些子模块、何时读哪个
├── tools/review.md   ← Tool 设计专项
├── safety/review.md  ← 安全边界专项
├── loop/review.md    ← 死循环风险专项
└── context/review.md ← 上下文管理专项
```

顶层 skill.md 的关键写法：
```markdown
## 判断规则
- 用户指定某个维度 → 只读对应子模块
- 用户说"全量"    → 按顺序读所有子模块
```

### 价值

```
平铺 skill：整个复杂 prompt 一次性进 context
层级 skill：顶层描述（小）→ 子模块按需加载 → 只用到的进 context
```

和软件工程模块化思想一致：高内聚，按需加载，关注点分离。

### ⚠️ Skill 的 Lazy Loading 机制

Session 开始时：
- Skill 的 `description`（frontmatter）加载进 context → Claude 知道有哪些 skill 可用
- Skill 全文**不加载**，用到时才展开

触发方式：
| 模式 | 触发方 |
|------|--------|
| 默认（有 description） | Claude 自动判断 **或** 人工 `/xxx` |
| `disable-model-invocation: true` | 只能人工 `/xxx`，描述也不加载，context 成本为零 |

**→ description 写得好不好直接影响 Claude 能否正确判断何时自动调用。**

### ⚠️ Skill 的副作用：约束越强，越容易降智

实测对比：同一个项目，用 `my-agent-audit`（固定4维度）vs 不用 skill 直接问：

```
用了 skill：
  - 只看4个预设维度
  - 没发现 config.py 明文 API Key（严重问题）
  - 没发现 Sub-agent 重复工具定义（DRY 问题）
  - 没有整体架构分析

没用 skill：
  - 开放探索，发现了更多问题
  - 有架构总结、设计模式分析
  - 输出质量更高
```

**根本原因：** Skill 用固定维度约束了 Claude，skill 没覆盖到的问题 Claude 不会主动提。

### 什么时候用 Skill，什么时候直接问

| 场景 | 建议 |
|------|------|
| 第一次全量探索、调试未知问题 | 直接问，不用 skill |
| 需要稳定一致的输出格式 | 用 skill |
| 专项复查（只看安全/只看循环） | 用 skill |
| 自动化流水线 | 用 skill |

### ⚠️ 不要无脑囤积 Skills

- 大量 skill 的 description 全部加载进 session context，占用 token
- 固定维度的 skill 可能比 Claude 通用能力更弱，反而降智
- 针对旧模型调优的 skill，新模型出来后可能变成噪音
- **按需取用，遇到具体需求再写/找，不要提前囤积**
